# Boomerang — правила проекта

Telegram-бот «второй мозг»: пользователь пересылает контент → бот возвращает его в нужный момент. Бот: `@Boomerang_bot`. Полная спека — в `boomerang-bot-spec.md`.

## Тезис (не нарушать)
Продукт продаёт **извлечение (retrieval), а не хранение (storage)**. «Сохранить» — фейковая продуктивность; Saved Messages = кладбище. Ценность в том, что накопленное **само возвращается** к пользователю. Если фича делает бота «местом для хранения», а не «механизмом возврата» — она против тезиса.

## Сквозные принципы (соблюдать в каждом решении)
1. **Ноль трения на входе.** Пользователь просто пересылает. Никаких обязательных тегов, форм, шагов. Любой обязательный ручной шаг — стоп, это против продукта.
2. **Классифицируй по самому дешёвому информативному сигналу.** Лесенка: текст подписи / OG-мета / имя файла / первые ~200–500 символов → и только потом, по запросу, глубокое чтение. Не лезь глубже без необходимости.
3. **Дорогое — только по запросу.** Глубокое чтение файла/видео — при вопросе по нему, не при сохранении.
4. **Семантика — основа поиска, категории — вспомогательный recall.** Извлечение строится на эмбеддингах (вектор), а НЕ на тегах. Но поиск гибридный: если запрос похож на имя кластера («животные» → «Животные»), подмешиваем записи этой категории — иначе запрос «по названию категории» промахивается по посту, где тема растворена (кот в политическом посте). Категория лишь добавляет recall, не заменяет вектор и не становится фундаментом. Промах авто-тега НЕ критичен. Сопоставление запроса с именами кластеров — триграммное (терпит русскую морфологию), порог семантики низкий: шум отсекаем порядком + лимитом, а не высоким порогом.
5. **Категории всплывают снизу (кластеризация).** Не хардкодить список категорий. Имена кластеров — через LLM, 1 вызов на кластер, не на сообщение.
6. **Микро-ценность мгновенно.** На каждую пересылку — мгновенное «принял», затем редактирование сообщения с авто-классификацией (Level 1).

## Обработка контента (кратко)
- **Ссылки/статьи:** title + OG-мета, НЕ читать тело. Ядро, дёшево.
- **Текстовые посты:** текст уже в сообщении.
- **Документы (Word/PDF 2–10 стр.):** редкий дорогой край; читать целиком ок (или лениво по запросу).
- **Картинки:** OCR (tesseract.js, с гейтом уверенности — мусор с фото без текста отсекается) + vision-аннотация в L2 (gpt-4o-mini, detail:low, один вызов: описание + категория + заголовок). Описание — только в индекс (**сырой OCR/description не показывать пользователю**), юзеру — заголовок; категория — сид тематического кластера, фолбэк — полка «Изображения». OCR и vision дополняют друг друга: OCR — дословные буквы в full-res, vision — смысл.
- **Голосовые/аудио/видео:** транскрипция аудио(дорожки) в L2 (STT через Groq whisper, дёшево; STT_API_KEY обязателен, как и все поля .env). **Сырой транскрипт не показывать** — только в индекс (как OCR); пользователю — LLM-заголовок одной строкой. Файлы >20MB Bot API ботам не отдаёт — такие остаются с подписью/тегами, юзера честно предупреждаем.

## Архитектура (3 уровня)
- **L1 синхронно:** «принял» сразу → дешёвая классификация → edit сообщения.
- **L2 фон (очередь):** эмбеддинги, OCR, отнесение к кластеру.
- **L3 по требованию:** глубокое чтение файла/видео при вопросе.

## Стек
TypeScript (strict) · grammY (+ runner) · PostgreSQL + pgvector · Drizzle · дешёвые модели для эмбеддингов и LLM-задач · tesseract.js · open-graph-scraper · pg-boss (очередь фоновых задач на Postgres; Redis — позже, только по необходимости).

## Приоритет разработки
Извлечение (поиск + LLM-синтез, режим 1) тащить вперёд как можно раньше — на нём проверяется весь тезис. Не «список ссылок», а **связный синтез со ссылками на источники**.

## Конфиг
Все поля `.env` обязательны, фолбэков и опциональных фич нет (пустое поле = ошибка на старте). Один `OPENAI_API_KEY` на LLM + эмбеддинги + vision; `STT_API_KEY` — Groq. Базовые URL и имена моделей — константы в `src/ai/*`; пороги/цены — `src/config/tuning.ts` (env-override опционален).

## Чего НЕ делать в MVP
Видео-контент глубже транскрипции аудиодорожки (анализ кадров, файлы >20MB через локальный Bot API server — отложено); проактивное всплытие (после MVP, риск спама); платежи; веб-интерфейс; экспорт/расшаривание.
