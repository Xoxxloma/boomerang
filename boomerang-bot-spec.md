# Boomerang — Telegram «второй мозг» (спецификация для разработки)

> **Boomerang** — финальное имя. Бот в Telegram: `@Boomerang_bot`. Метафора прямо объясняет продукт: кидаешь контент → он возвращается к тебе в нужный момент (а не тонет в кладбище Saved Messages). Документ описывает продукт, архитектурные решения и MVP-скоуп. Стек: TypeScript + grammY. Писалось как ТЗ для разработки (в т.ч. через Claude Code).

---

## 1. Что строим и зачем (продуктовый тезис)

«Сохранить» — это фейковая продуктивность. У всех Saved Messages в Telegram — это кладбище: туда кидают статью/мысль/мем, получают микро-дофамин «я что-то сделал», и больше никогда не возвращаются. Мест для хранения навалом (Notion, Obsidian, те же Saved Messages). Боль не в хранении — боль в том, что **сохранённое не возвращается к человеку само**.

Поэтому продукт продаёт **извлечение (retrieval), а не хранение (storage)**. И он **активный**, а не пассивный: бот не ждёт, пока ты придёшь искать, — он сам всплывает с релевантным. Это то, чего не делают ни Saved Messages, ни PKM-приложения.

Строим продукт для многих пользователей. Основатель — первый пользователь и тестировщик: проверяем на собственной боли, но архитектура и решения изначально рассчитаны на multi-user (изоляция данных по пользователю, персональная таксономия у каждого).

### Чем отбиваемся от конкурентов
- **Saved Messages + нативный поиск** — главная «достаточно хорошая» альтернатива. Бьём в то, что это свалка без структуры и без ответов на вопросы. Если бот реально *отвечает* «что я сохранял про ипотеку» синтезом, а не списком ссылок — разница ощутима. Если он просто чуть лучше ищет по ключевым словам — нас съест нативный поиск. **Ценность строго в извлечении, а не в хранении.**
- **Conoted** — непрямой конкурент, целит в бизнес (сбор контактов из каналов). Ниша «чисто личное, ноль трения, внутри Telegram» им не занята.
- **DIY-шаблоны (n8n, ChatGPT-связки)** — для технарей. Обычный пользователь не соберёт. Наша задача — упакованный продукт.
- **Ров (moat):** накопление данных (чем дольше пользуешься — тем дороже уходить) + персональная таксономия, которую бот выучивает под конкретного человека.

## 2. Сквозные принципы (правят каждым техническим решением)

1. **Ноль трения на входе.** Пользователь просто пересылает контент. Ничего не обязан тегировать, заполнять, нажимать. Любой обязательный ручной шаг убивает смысл (иначе проще открыть Notion).
2. **Классифицируй по самому дешёвому информативному сигналу, который есть в сообщении.** Не лезь глубже без необходимости. Это единая лесенка для всех типов контента (см. §3).
3. **Дорогое — только по запросу.** Глубокое чтение файла/видео происходит, только когда по нему реально задали вопрос, а не при сохранении.
4. **Теги — это навигация и ощущение порядка для человека, НЕ механизм поиска.** Поиск идёт по эмбеддингам (семантика). Значит, промах авто-тега не критичен — найдётся всё равно. Это сильно снижает планку точности классификации.
5. **Категории всплывают снизу (кластеризация), а не задаются сверху.** Навязанная таксономия бесит. У каждого своя нарезка мира.
6. **Микро-ценность на каждую пересылку, мгновенно.** Каждое сохранение не должно быть броском в пустоту — это и есть лечение исходной боли.

## 3. Типы контента и обработка (лесенка дёшево → дорого)

Реальный микс пользователя (по факту, в порядке частоты): **статьи и посты со ссылками → мемы/картинки → редко документы (Word/PDF 2–10 стр.)**.

### 3.1 Ссылки на статьи / внешние посты — ЯДРО, самый дешёвый случай
- НЕ читаем тело статьи. Тянем `<title>` + Open Graph мету (`og:title`, `og:description`, `og:image`).
- Классификация по заголовку + описанию — копейки, мгновенно.
- На статьях извлечение работает максимально красиво: «что я сохранял про X» даёт реальный синтез.

### 3.2 Текстовые посты Telegram — тоже дёшево
- Текст уже в самом сообщении, читать неоткуда больше нечего. Берём как есть.

### 3.3 Документы (Word / PDF, 2–10 страниц) — редкий «дорогой край»
- Редкие → даже если обработаем чуть дороже/медленнее, на общую экономику не влияет.
- 2–10 страниц — это не диссертация, по токенам терпимо. Можно читать целиком при сохранении (или лениво — по первому вопросу, см. §5 Level 3).

### 3.4 Картинки (мемы, скриншоты) — отдельная единая полка, БЕЗ понимания смысла
- **Не понимаем смысл картинки.** Никакой vision-модели, никакого «что значит этот мем». Дорого, ненадёжно, не окупается. Отрезано принципиально.
- Одна категория «Изображения», не бьём на подкатегории.
- **НО: прогоняем OCR**, чтобы внутри работал поиск (иначе папка — то же кладбище, просто переехавшее). OCR ≠ понимание; это механическое считывание букв. Дёшево и надёжно.
- Для нашего контента (мемы со словами + скрины) текст есть почти всегда → OCR попадает почти стопроцентно.
- **Сырой OCR-текст НЕ показываем пользователю** (там бывает каша `1l0O`). Он лежит только в индексе, под капотом, и невидимо помогает поиску. Грязный результат не портит UX.
- Бессловесная фотка → пустой OCR → просто лежит в «Изображениях» без текстового индекса. Это ожидаемое поведение, не баг.

### 3.5 Пост с видео — связка «текст + видеофайл», обрабатываем раздельно
- Смысл обычно в **подписи к посту**, а не в видео. Берём текст подписи как обычный пост (дёшево) — этого хватает в большинстве случаев.
- **Само видео по умолчанию НЕ обрабатываем.** Понять видео = достать аудио → Whisper → может, кадры. Дорого, медленно, чаще всего не нужно (суть была в подписи). Сохраняем как объект + превью.
- Голое видео без подписи → как картинка без текста: кладём в медиа-полку, искабельным не делаем.
- (Опционально, на потом) короткое видео без подписи → прогнать аудио через Whisper, переиспользуя движок голосовых. Строго по триггеру, не на каждое автоматом.

## 4. Два экзистенциальных риска и их решения

### Риск 1 — холодный старт (пустая база бесполезна, а первое впечатление решает)
**Решение:** при подключении — массовый импорт существующих Saved Messages / выбранного канала, чтобы сразу был корпус и первый же запрос что-то находил.

Пайплайн разовый, мгновенности не требует (минуту подождать ок):
1. Прогнать все сообщения через **эмбеддинги** (дёшево, эмбеддинг ≪ генерации).
2. **Кластеризовать векторы** (бесплатно, чистая математика, ноль токенов).
3. LLM-вызов **на кластер** (не на сообщение!) — дать человеческое имя: «похоже, тут про ипотеку и ставки». 800 сохранёнок в 15 кластерах = 15 вызовов, не 800.

**Wow-момент онбординга:** бот не просто разложил по полкам — он **показал человеку его самого**: «за год ты сохранил 60 штук про ИИ-инструменты и 40 про переезд». Цепляет ещё до первого запроса.

### Риск 2 — а будет ли вообще достаточно пересылать?
**Решение:** микро-ценность мгновенно на каждый бросок (см. §5 Level 1) + авто-тег вместо ручного.

**Авто-тег + правка в один тап.** НЕ «пользователь обязан тегировать», а «бот предлагает тег сам, пользователь одним тапом исправляет при промахе». Трения ноль (можно игнорить). **Каждая правка учит систему личной нарезке пользователя** — это и есть ров. Поиск всё равно идёт по эмбеддингам, так что промах тега не ломает извлечение.

## 5. Архитектура обработки — три уровня (чтобы UX не тормозил)

**Level 1 — мгновенно (синхронно).**
Бот сразу отвечает «принял». Затем по дешёвым сигналам (URL, заголовок/OG, имя файла, первые ~200–500 символов) маленькая модель классифицирует и **редактирует своё же сообщение** → «положил в „Недвижимость“, связано с тем, что ты кидал про ставки месяц назад». В Telegram сообщения редактируются — выглядит, будто бот «подумал». Это и есть микро-ценность на каждую пересылку.

**Level 2 — фон (асинхронно, очередь).**
Тяжёлое индексирование: эмбеддинги, поиск связей с прошлым, OCR картинок. Пользователь не ждёт.

**Level 3 — по требованию.**
Глубоко читаем файл/видео, только если по нему задали вопрос.

## 6. Извлечение — собственно ценность продукта

Три режима «возвращать накопленное»:

1. **Ответ по запросу.** «Что я сохранял про ипотеку?» → **синтез**, а не список ссылок. Семантический поиск по эмбеддингам → топ-N релевантных → LLM собирает связный ответ со ссылками на источники.
2. **Проактивное всплытие (главный дифференциатор).** «Две недели назад ты сохранял вот это про X, сейчас релевантно». Решает удержание: боту не нужно, чтобы ты вспомнил вернуться. **Риск — спам; это право надо заслужить.** В MVP — осторожно/выключено по умолчанию, включаем после того, как накопится сигнал.
3. **Синтез-дайджест.** «За неделю накидал 12 ссылок — вот 3 темы, которые тебя зацепили».

Приоритет ценности: **режим 1 обязателен в MVP** (это ядро), режим 3 — лёгкое добавление, режим 2 — самый сильный, но и самый рискованный, аккуратно после MVP.

## 7. Категории через кластеризацию (детали)

- Категории НЕ захардкожены. Появляются из кластеризации эмбеддингов.
- При импорте — кластеризуем разом, именуем кластеры через LLM (1 вызов/кластер).
- При новом сообщении — относим к ближайшему существующему кластеру по косинусной близости; если далеко от всех — заводим кандидата на новую категорию.
- Правки пользователя (переложил из A в B) корректируют отнесение и учитываются дальше.

## 8. Технический стек (JS/TS)

| Слой | Выбор | Заметки |
|---|---|---|
| Язык | **TypeScript** | строгий режим |
| Bot framework | **grammY** | + `@grammyjs/runner` для конкурентной обработки, `conversations` плагин при необходимости |
| Хранилище | **PostgreSQL + pgvector** | один движок для данных и векторного поиска. Альтернатива для локального MVP: SQLite + `sqlite-vec` |
| ORM/доступ | **Drizzle** (или Prisma) | Drizzle легче ложится на pgvector |
| Эмбеддинги | дешёвая модель эмбеддингов (напр. `text-embedding-3-small`) | копейки за объём; цены перепроверить |
| LLM (классификация/именование/синтез) | дешёвая модель (`gpt-4o-mini` / Claude Haiku класс) | задачи простые, топовая не нужна |
| OCR | **`tesseract.js`** (или системный `tesseract` через child_process) | старт локально/бесплатно; на скринах отлично, на «диких» мемах средне → при промахах переключиться на облачный OCR (замена одной функции) |
| Метаданные ссылок | `open-graph-scraper` или `cheerio` + fetch | тянем title + OG |
| Голос/видео-аудио (опц.) | Whisper API | переиспользуется для голосовых и для опц. видео |
| Очередь фоновых задач | **pg-boss** (на Postgres) | durable-очередь без отдельной инфры; Redis/BullMQ — позже, если упрёмся в throughput |
| Платежи (позже) | Telegram Stars / YooKassa | не в MVP |

**Принцип по OCR:** не угадывай качество — проверь Tesseract на *своём реальном* контенте, при промахах меняй на API.

## 9. Модель данных (черновик)

```
items
  id              uuid pk
  user_id         bigint                 -- tg user id
  source_chat     text                   -- откуда переслано (опц.)
  type            enum(link|tg_post|document|image|video|text|voice)
  raw_text        text                   -- текст поста / подписи / своя мысль
  url             text                    -- для ссылок
  title           text                    -- title / og:title
  description     text                    -- og:description
  ocr_text        text                    -- ТОЛЬКО под капотом, не показывать
  transcript      text                    -- для voice/опц. видео
  file_path       text                    -- хранилище файла/превью
  embedding       vector                  -- pgvector
  cluster_id      uuid fk -> clusters     -- текущая категория
  cluster_locked  bool                    -- true если пользователь правил вручную
  created_at      timestamptz
  indexed_at      timestamptz             -- когда прошёл Level 2

clusters
  id              uuid pk
  user_id         bigint
  name            text                    -- человеческое имя от LLM
  centroid        vector                  -- для отнесения новых item
  size            int
  updated_at      timestamptz

users
  id              bigint pk               -- tg user id
  created_at      timestamptz
  import_done     bool
  settings        jsonb                   -- напр. proactive_surfacing: off
```

Индексы: HNSW/IVF на `items.embedding` и `clusters.centroid` (pgvector).

## 10. Скоуп MVP

**В MVP входит:**
- Приём пересланного: ссылки, текстовые посты, документы (2–10 стр.), картинки, посты с видео (берём подпись).
- Level 1: мгновенное «принял» + авто-классификация с редактированием сообщения.
- Level 2 (фон): эмбеддинги, OCR картинок, отнесение к кластеру.
- Массовый импорт Saved Messages на онбординге + кластеризация + wow-сводка «вот ты какой».
- Авто-тег + правка в один тап (кнопка «не туда → выбрать другую категорию»).
- **Извлечение, режим 1:** «что я сохранял про X» → синтез со ссылками.
- Семантический поиск по эмбеддингам (включая `ocr_text` и `transcript`).

**Явно НЕ в MVP (отложено):**
- Понимание смысла картинок (vision) — отрезано принципиально.
- Обработка видео-контента (только подпись; аудио через Whisper — опционально, потом).
- Проактивное всплытие (режим 2) — после MVP, осторожно, из-за риска спама.
- Платежи, командные тарифы, мультиюзер-онбординг.
- Экспорт, расшаривание, веб-интерфейс.

## 11. Порядок сборки (вехи)

1. **Каркас:** grammY-бот, приём сообщения, эхо «принял», БД, таблицы.
2. **Ссылки + текст:** OG-парсинг, сохранение item, эмбеддинг, запись в индекс.
3. **Поиск/извлечение (режим 1):** команда/запрос → семантический поиск → LLM-синтез. ← как можно раньше, это ядро ценности, на нём проверяется весь тезис.
4. **Категории:** кластеризация, именование, отнесение новых, кнопка-правка.
5. **Картинки + OCR (фон, Level 2):** tesseract.js, `ocr_text` в индекс (не показывать).
6. **Документы:** парсинг Word/PDF, ленивое глубокое чтение по запросу (Level 3).
7. **Импорт Saved Messages + wow-сводка онбординга.**
8. **Дайджест (режим 3).**
9. (после MVP) Проактивное всплытие, видео-аудио, платежи.

> Совет: п.3 (извлечение) тащи вперёд как можно раньше — если синтез по маленькому корпусу уже ощущается полезным, тезис подтверждён; если нет — лучше понять это до того, как вложиться в импорт и кластеризацию.

## 12. Открытые вопросы / решить по ходу
- **Видео без подписи:** в MVP игнор; решить позже, гонять ли короткие через Whisper и по какому триггеру.
- **Доля бессловесных фоток** в реальном контенте — влияет, насколько окупается OCR (если фоток много, OCR можно отложить).
- **Порог «далеко от всех кластеров»** для заведения новой категории — подобрать эмпирически.
- **Когда включать проактивное всплытие** и как не превратить в спам (частота, «заслуженность»).
- Хранилище: остаёмся на Postgres + pgvector (одна БД на данные, векторный поиск и очередь pg-boss).

---

## 13. Заметки по экономике
- Ядро потока (ссылки, посты, картинки) — дешёвая «норма»: эмбеддинг + классификация по дешёвым сигналам = копейки.
- Дорогой край (документы, видео) — редкий и/или ленивый (по запросу), на общую стоимость не давит.
- Эмбеддинги ≪ генерации; кластеризация бесплатна; именование — 1 вызов/кластер.
- OCR на старте локальный (бесплатно). Whisper — только по необходимости.
- Все цены провайдеров **перепроверять перед стартом** — меняются.
