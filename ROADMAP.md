# Boomerang — Roadmap

Что **не** входит в MVP v0.1 и почему. Тезис продукта не нарушать: продаём **извлечение, а не хранение** (см. `CLAUDE.md`, `boomerang-bot-spec.md`).

## v0.1 (текущий MVP)
Приём пересланного (ссылки, текст, документы, картинки, видео-подпись) · L1 «принял» + авто-классификация · L2 (эмбеддинги, OCR, кластеры) · извлечение режим 1 (синтез со ссылками) · категории через кластеризацию + правка в один тап · дайджест (режим 3).

Порядок вех — в `boomerang-bot-spec.md` §11 и в плане `~/.claude/plans/abstract-launching-octopus.md`.

## После v0.1 (бэклог)

| Фича | Почему отложено | Когда |
|---|---|---|
| **Импорт Saved Messages + wow-сводка онбординга** | Bot API не видит чужие Saved Messages → нужен MTProto/GramJS userbot + auth по номеру телефона. Главный ответ на холодный старт (риск 1, §4). | Отдельная веха сразу после v0.1 |
| **Проактивное всплытие (режим 2)** | Самый сильный дифференциатор, но риск спама; «право надо заслужить». По умолчанию off (`users.settings.proactive_surfacing`). | После накопления сигнала |
| **Видео-аудио через Whisper** | Дорого/медленно, суть обычно в подписи (§3.5). Строго по триггеру, не автоматом. | Опционально, по необходимости |
| **Платежи** (Telegram Stars / YooKassa), тарифы, онбординг команд | Не в скоупе раннего релиза. | Позже |
| **Экспорт / расшаривание / веб-интерфейс** | Не относится к ядру (извлечение внутри Telegram). | Позже |

## Рефакторинг к multi-user / прод

Проект — multi-user. Что уже сделано и что осталось вынести из «одно-процессных» допущений.

**Сделано (A1):** состояние вынесено из памяти процесса в Postgres — очередь L2 на **pg-boss**
(durable, ретраи), буфер альбомов и сессия правки категории — в таблицах. Бот переживает рестарт и
допускает несколько инстансов-воркеров.

**Осталось:**
| Пункт | Что не так | Когда |
|---|---|---|
| **A2 — намерение «вопрос/заметка»** | `looksLikeQuery` угадывает регексом (хрупко). Нужны явные режимы/кнопки «Спросить/Сохранить» или LLM-классификатор намерения. | До роста аудитории |
| **A3 — горизонтальное масштабирование** | Сейчас один инстанс на long-polling. Перейти на вебхуки + несколько stateless-инстансов (состояние уже общее после A1). | При росте нагрузки |
| **A4 — лимиты/квоты OpenAI per-user** | Один пользователь может выжрать бюджет. Нужны rate-limit, квоты, мониторинг расходов. | До публичного запуска |
| **B — тех-долг** | Ленивое L3-чтение документов (сейчас eager в L2); пороги 0.45/0.28 в конфиг; пул OCR-воркеров. | По мере необходимости |
| **Redis / кэш** | Вводим только по конкретной причине (кэш, rate-limit на больших оборотах, pub/sub), не «на всякий случай». | При throughput >~1k задач/сек |

### Принципиально НЕ делаем
- **Vision-понимание смысла картинок** — дорого, ненадёжно, не окупается (§3.4). Картинки = одна полка + OCR для искабельности, без понимания смысла. Не возвращаемся к этому.

## Открытые вопросы (решить по ходу, §12)
- Порог «далеко от всех кластеров» для заведения новой категории — подобрать эмпирически на своём корпусе.
- Качество `tesseract.js` на **реальном** контенте; при промахах — свап на облачный OCR (замена одной функции).
- Доля бессловесных фоток — влияет, окупается ли OCR.
- Когда и как включать режим 2, чтобы не превратить в спам (частота, «заслуженность»).
- Хранилище: остаёмся на Postgres+pgvector (решено для v0.1).
