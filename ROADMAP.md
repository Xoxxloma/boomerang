# Boomerang — Roadmap

Что **не** входит в MVP v0.1 и почему. Тезис продукта не нарушать: продаём **извлечение, а не хранение** (см. `CLAUDE.md`, `boomerang-bot-spec.md`).

## v0.1 (текущий MVP)
Приём пересланного (ссылки, текст, документы, картинки, видео-подпись) · L1 «принял» + авто-классификация · L2 (эмбеддинги, OCR, кластеры) · извлечение режим 1 (синтез со ссылками) · категории через кластеризацию + правка в один тап · дайджест (режим 3).

Порядок вех — в `boomerang-bot-spec.md` §11 и в плане `~/.claude/plans/abstract-launching-octopus.md`.

## Рефакторинг к multi-user / прод

Проект — multi-user. Что уже сделано и что осталось вынести из «одно-процессных» допущений.

**Сделано (A1):** состояние вынесено из памяти процесса в Postgres — очередь L2 на **pg-boss**
(durable, ретраи), буфер альбомов и сессия правки категории — в таблицах. Бот переживает рестарт и
допускает несколько инстансов-воркеров.

**Сделано (A2):** намерение «вопрос/заметка» решено явной маршрутизацией, без угадывания. Поиск — только
по `/find`, кнопке 🔍 Найти и ответу на приглашение поиска (`bot/handlers/search.ts`); любой иной текст
уходит в приём (сохраняется). Эвристики `looksLikeQuery` больше нет.

**Сделано (A4):** per-user квоты + бюджет-гард. Учёт расхода (`ai/usage.ts`, `db/usage.ts`), breaker
normal/degraded/paused, персональный дневной потолок, админ-алерты при сбоях (`bot/alerts.ts`),
тесты (`test/usage.test.ts`, `test/alertThrottle.test.ts`, `test/process-idempotency.test.ts`).
Остаточные LOW-пункты бюджета — ниже, в «Известные ограничения и тех-долг».

**Сделано (A5):** ручной импорт корпуса через JSON-экспорт Telegram (`/import`) + всплеск-детектор:
дедуп, батч-эмбеддинги, батч-кластеризация, wow-сводка (`import/burst.ts`, `import/batch.ts`,
`import/draft.ts`, `import/progress.ts`). Без трения (авто-импорт через userbot) — см. таблицу выше.

**Сделано (режим 2 — проактивное всплытие):** созревание (maturity) + резонанс реализованы
(`retrieval/proactive.ts`, `db/surfacing.ts`) — opt-in, дневной кап, кулдаун. **По умолчанию выключен**
(`users.proactiveMode`); включать осторожно по накоплении сигнала (риск спама, §6).

**Осталось:**
| Пункт | Что не так | Когда |
|---|---|---|
| **A3 — горизонтальное масштабирование** | Сейчас один инстанс на long-polling. Перейти на вебхуки + несколько stateless-инстансов (состояние уже общее после A1). Снимает и LOW-ограничение «один инстанс» (см. ниже). | При росте нагрузки |
| **B — тех-долг** | Ленивое L3-чтение документов (сейчас eager в L2); пул OCR-воркеров. Детали — ниже, в «Известные ограничения и тех-долг». | По мере необходимости |
| **Redis / кэш** | Вводим только по конкретной причине (кэш, rate-limit на больших оборотах, pub/sub), не «на всякий случай». | При throughput >~1k задач/сек |

### Принципиально НЕ делаем
- **Vision-понимание смысла картинок** — дорого, ненадёжно, не окупается (§3.4). Картинки = одна полка + OCR для искабельности, без понимания смысла. Не возвращаемся к этому.

## Известные ограничения и тех-долг (LOW)

Осознанно отложенное: LOW-severity и «потом, по необходимости». Срочное чинится сразу и сюда не попадает.

### Бюджет-гард (защита от спама и денежных трат)

Аудит дневного учёта расхода (июнь 2026). Критичное и среднее — починено (см. «Сделано (A4)» и коммиты по
`usage`/`alerts`/`import`). Ниже — осознанно отложенный LOW, по убыванию важности.

- **[LOW] Один инстанс бота — обязателен.** Флаш учёта перезаписывает строку `usage_daily`
  (`onConflictDoUpdate set = excluded.*`, память = источник истины — [src/db/usage.ts](src/db/usage.ts)).
  Два+ инстанса затрут счётчики друг друга → глобальный расход недосчитается → breaker не сработает.
  Пока деплой одно-инстансный — ок. Если когда-нибудь масштабироваться: перейти на **аддитивный флаш по
  дельтам** (`set cost = usage_daily.cost + excluded.delta`, флашить прирост с последнего флаша), а не
  перезапись. До тех пор — НЕ запускать вторую реплику процесса бота. Системное снятие ограничения —
  переход на вебхуки + stateless-инстансы («A3 — горизонтальное масштабирование» выше).

- **[LOW] Краш-окно учёта ≤ 60с.** Флаш раз в 60с ([src/index.ts](src/index.ts), `USAGE_FLUSH_INTERVAL_MS`).
  При жёстком крэше теряются траты с последнего флаша → после рестарта лёгкий недосчёт (fail-open, ограничено
  одной минутой; на SIGINT/SIGTERM есть финальный флаш). Уменьшать интервал/флашить синхронно — не стоит ради
  этого нагружать БД. Оставить как есть.

- **[LOW] `degraded` не режет `parseQuery`.** При глобальном `degraded` дорогой синтез на `/find` уже
  пропускается ([src/bot/handlers/search.ts](src/bot/handlers/search.ts)), но LLM-разбор запроса `parseQuery`
  всё ещё вызывается. Это небольшой LLM-вызов (≤256 токенов). Можно в degraded пропускать parseQuery
  (fallback на passthrough), если глобальная нагрузка станет проблемой.

- **[LOW] `String(costUsd)` в научной нотации.** Микро-суммы вида `2e-7` уходят в `numeric(12,6)` и
  округляются до 0 (теряются субмикродоллары). Безвредно при текущих ценах. Если важна точность учёта на
  уровне отдельных дешёвых вызовов — флашить в микродолларах целым числом.

- **[LOW] Цены LLM/эмбеддингов захардкожены в tuning-дефолтах.** [src/config/tuning.ts](src/config/tuning.ts)
  (`llmPricePromptPer1k` и т.д.) — при смене модели/провайдера (константы в `src/ai/*`) учёт «поедет».
  Уже есть TODO «СВЕРИТЬ с актуальным прайсом». При смене модели — обновлять цены в tuning.ts.

- **[LOW] Пробелы в тестах учёта.** Нет интеграционных тестов на `db/usage.ts` (флаш/регидрация — нужен живой
  Postgres). Покрыто юнит-тестами: ядро `usage.ts`, троттлинг алертов, дубль-эмбеддинг `processItem`,
  бюджет-стоп заливки.

- **[LOW] Дневное окно анти-спама (проактивные всплытия) — не UTC.** `countSurfacedToday`
  ([src/db/surfacing.ts:34](src/db/surfacing.ts#L34)) считает «сегодня» через `date_trunc('day', now())` —
  это **TZ сессии Postgres**, а не UTC (хотя докстринг говорит «UTC», и бюджет-ядро живёт по UTC). Если TZ
  подключения не UTC — лимит `PROACTIVE_DAILY_CAP` сбрасывается по локальной полуночи, рассинхрон на часы.
  Влияния на деньги/данные нет (это лишь анти-спам потолок проактивных сообщений). Кандидат на быстрый
  фикс (не «отложено навсегда»): привести к UTC — `now() AT TIME ZONE 'UTC'` или сравнение с UTC-границей
  из кода.

### Тех-долг реализации (не бюджет)

Детали к строке «B — тех-долг» выше.

- **[LOW] Ленивое L3-чтение документов.** Сейчас документы читаются **eager в L2**
  ([src/queue/jobs/process.ts:32-39](src/queue/jobs/process.ts#L32)) — при индексировании, а не по
  первому вопросу. Спека §5 (Level 3) предполагает глубокое чтение только по запросу. Документы редки,
  на экономику не давит → отложено. Перевести на ленивое чтение, когда дорогой край начнёт мешать.

- **[LOW] Пул OCR-воркеров.** Tesseract-воркер — один ленивый singleton
  ([src/content/ocr.ts:8-16](src/content/ocr.ts#L8)). При росте потока картинок OCR станет узким местом
  (всё сериализуется через один worker). Завести пул, когда упрёмся в throughput картинок.

## Открытые вопросы (решить по ходу, §12)
- Порог «далеко от всех кластеров» для заведения новой категории — подобрать эмпирически на своём корпусе.
- Качество `tesseract.js` на **реальном** контенте; при промахах — свап на облачный OCR (замена одной функции).
- Доля бессловесных фоток — влияет, окупается ли OCR.
